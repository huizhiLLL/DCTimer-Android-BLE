# MoYu32 智能魔方计时一期方案

## 0. 当前状态更新（2026-04-09）

这份文档最初只覆盖 `MoYu32` 一期接入方案，但当前代码状态已经继续向前推进了一步：

- 已在原有 `MoYu32` 兼容基础上，补充 `QiYi / Tornado V4 智能版` 的蓝牙协议接入。
- `QiYi / Tornado V4` 当前沿用现有 `SmartCube` 计时链路，支持扫描、连接、初始状态同步、转动上报、电量同步，以及“标记已打乱 -> 首转开表 -> 还原停表”这条主流程。
- Android 侧实现参考 `ref/cstimer/src/js/hardware/qiyicube.js`，采用独立 `QiyiCubeProtocol` 接入，而没有继续把协议细节堆进旧的通用分支里。
- 当前 `MoYu32` 与 `QiYi` 两条新协议链已经统一抽成 `SmartCubeProtocol` 接口，便于后续继续扩展更多智能魔方。
- 蓝牙设备选择弹窗顶部原有的协议类别切换已经移除，当前改为直接列出扫描结果，并在连接与服务发现阶段自动识别设备类型。
- 已继续参考 `ref/cstimer/src/js/hardware/gancube.js`，补充 `GAN` 主流智能魔方协议接入，当前覆盖 `v2 / v3 / v4` 三条主流链路。
- 旧的 `GAN Timer / Giiker / 老 GANi` 解析分支已退出当前扫描连接主链，只保留代码备份，不再参与当前产品逻辑。
- 工程底座已升级到 `AndroidX + AGP 8.9.2 + Gradle 8.11.1 + JDK 17`，当前 `compileSdk / targetSdk` 已提升到 `35`。
- 文件导入导出与背景图选择已切换到 `SAF / Uri` 主链，旧 `FileSelectorDialog` 已退出当前主流程。
- 主线 APK 自更新入口、安装权限与 `FileProvider` 已移除，当前工程按正式发布口径收口。
- 智能魔方 3D 状态图已从静态三面位图升级为自定义立体渲染控件，当前支持更接近实体魔方的固定视角、明暗分层和单步转层补间动画。

也就是说，下面正文仍然保留一期 `MoYu32` 设计背景，但阅读时需要以当前代码优先：当前已不再是“只支持 MoYu32”，而是至少支持 `MoYu32 + QiYi/Tornado V4` 两条智能魔方协议链。

## 1. 背景

当前 DCTimer 仓库中已经存在智能魔方计时的半成品能力：

- 已有蓝牙扫描、连接、断开、设备列表 UI。
- 已有 `SmartCube` 模型，可根据转动累计时间、步数、步序。
- 已有智能魔方计时回调，可在“已打乱”后首转开表、在还原后停表并保存成绩。
- 数据库结果表已预留 `moves` 字段，可记录复原步序。

但现有实现主要围绕旧的 `Giiker / GANi / GAN Timer` 原型展开，协议层、设备层、计时层耦合较重，不适合继续直接堆叠新设备。

本方案面向第一期，仅新增 `MoYu32` 系列智能魔方的 `3x3` 计时支持。

## 2. 一期目标

### 2.1 目标范围

- 仅支持 `MoYu32` 系列智能魔方。
- 仅支持 `3x3`。
- 支持蓝牙扫描、连接、断开。
- 支持获取并显示当前魔方状态与电量。
- 支持“标记已打乱”后首转开表。
- 支持还原完成后自动停表。
- 支持保存成绩与复原步序。
- 支持 MAC 自动获取失败时手动输入 MAC。

### 2.2 非目标

- 不做多设备通用抽象的完全重构。
- 不做训练模式、连续训练、自动分段、复盘分析。
- 不做完整的物理级虚拟魔方引擎或任意视角交互，仅做轻量实时转层补间与固定观察视角优化。
- 不做 `2x2 / 4x4+ / 异形` 设备支持。
- 不追求一期覆盖所有 Android 蓝牙边角兼容，只保证 Android 15 真机可用并为后续兼容预留结构。

## 3. 设计原则

- 正确性优先：先保证协议解析、开停表时机、成绩保存正确。
- 最小可用优先：先做单设备单项目闭环，而不是提早追求通用平台。
- 分层优先：避免继续把设备协议写死在 `BluetoothTools` 中。
- 可验证优先：每个阶段都要能在真机上用日志与实际转动验证。

## 4. 关键输入与假设

### 4.1 已知输入

- 参考实现来源：`ref/cstimer/src/js/hardware/moyu32cube.js`
- 设备命名前缀：`WCU_MY3`
- 用户已有 MoYu32 系列真机，可实际验证
- 用户接受手动输入 MAC 作为回退方案

### 4.2 当前假设

- Android 侧可通过 `BluetoothDevice.getAddress()` 获取足够稳定的地址；若解密验证失败，则回退为手动输入 MAC。
- 一期沿用现有 `SmartCube` 的计时模型：已打乱后首转开表，还原即停表。
- 一期不要求与 csTimer 的所有高级状态判断完全一致，只要求在真实 3x3 计时流程下行为可靠。

如果上述假设在真机验证中不成立，需要回到协议层调整，不直接扩大功能范围。

## 5. 参考实现解读

`moyu32cube.js` 的核心流程可概括为：

1. 发现目标设备，匹配前缀 `WCU_MY3`
2. 连接服务 `0783b03e-7735-b5a0-1760-a305d2795cb0`
3. 获取读写特征：
   - read: `...cb1`
   - write: `...cb2`
4. 通过设备 MAC 计算 AES key/iv
5. 发送查询命令：
   - `161` 查询设备信息
   - `163` 查询当前魔方状态
   - `164` 查询电量
6. 订阅通知，解析消息：
   - `161` 设备信息
   - `163` 初始状态
   - `164` 电量
   - `165` 转动事件
7. 根据 move event 中的 move、time offset、move count 还原转动序列与设备时间线
8. 将转动回调给上层计时逻辑

这说明 Android 端不需要“照搬整个 csTimer”，而是需要完整迁移这一条协议链。

## 6. 一期推荐架构

### 6.1 分层

建议将智能魔方一期能力拆成以下四层：

1. `BleScan / BleConnection` 层  
   负责扫描、连接、权限、通知订阅、GATT 生命周期。

2. `CubeProtocol` 层  
   负责某一类设备的协议编解码、命令发送、消息解析、状态恢复。

3. `SmartCubeSession` 层  
   负责把“设备事件”转换成“计时事件”：初始状态、标记已打乱、首转开表、还原停表、统计步数与步序。

4. `UI / Activity` 层  
   负责设备选择、状态展示、错误提示、成绩保存。

### 6.2 为什么不继续直接改 `BluetoothTools`

当前 `BluetoothTools` 同时承担了：

- 扫描与连接
- 协议选择
- 解密
- 消息解析
- 上层 UI 回调

如果直接把 `MoYu32` 再塞进去，会进一步放大以下问题：

- 一个类同时维护多个协议状态机，调试困难
- 后续适配新设备时持续增加耦合
- Android 12+ 权限升级时改动面过大

因此一期建议采用“保守重构”：

- 不必先推翻旧实现
- 但新接入的 `MoYu32` 走独立协议类
- `BluetoothTools` 仅作为旧能力兼容壳或逐步收口的过渡层

## 7. 一期模块设计

### 7.1 设备类型

新增一个设备类型，例如：

- `BLEDevice.TYPE_MOYU32_CUBE`

设备选择弹窗中新增一个选项，例如：

- `MoYu32`

### 7.2 协议类

新增 `Moyu32CubeProtocol`，建议职责如下：

- 识别目标服务和特征 UUID
- 建立通知监听
- 发送 `161 / 163 / 164`
- 保存 `deviceName / deviceMac / battery / moveCnt / prevMoveCnt / deviceTime / deviceTimeOffset`
- 解析 `161 / 163 / 164 / 165`
- 产出统一事件：
  - `onCubeReady(facelet)`
  - `onBatteryChanged(level)`
  - `onMove(move, deltaTime, facelet)`
  - `onProtocolError(reason)`

### 7.3 解密类

新增 `Moyu32Cipher`，不依赖外部 `aes.jar` 黑盒。

职责：

- 接收 MAC
- 计算 key/iv
- 使用标准 AES-128 实现加解密
- 提供：
  - `init(mac)`
  - `encode(byte[])`
  - `decode(byte[])`

设计要求：

- 与 `moyu32cube.js` 的 `getKeyAndIv / encode / decode` 行为对齐
- 不混入 BLE 逻辑
- 允许后续复用于其他使用相同方案的设备

### 7.4 计时会话层

建议新增一个轻量会话控制层，例如 `SmartCubeSessionController`。

职责：

- 接收协议层的初始状态与 move 事件
- 同步到 `SmartCube`
- 判断是否与当前打乱状态一致
- 触发：
  - 标记已打乱
  - 首转开表
  - 还原停表
- 不负责 BLE 细节

这样可以把“设备协议”和“计时业务”拆开，减少 `MainActivity` 中的复杂判断。

### 7.5 UI 层

一期 UI 仅做必要改动：

- 蓝牙设备选择框新增 `MoYu32`
- 连接成功后可查看当前状态与电量
- 当自动获取 MAC 失败或解密失败时，弹框允许手动输入 MAC

不在一期新增复杂设置页。

当前状态展示已经从早期的静态三面模板升级为自定义立体渲染控件，当前能提供更稳定的视角和转层补间；后续仍可继续细调配色、边角质感与投影细节。

## 8. MAC 方案

### 8.1 主路径

优先尝试使用 Android `BluetoothDevice.getAddress()` 作为 MAC。

原因：

- Android 原生 BLE 与 Web Bluetooth 环境不同
- 有机会直接拿到可用地址，省去广告包解析和手输流程

### 8.2 回退路径

如果出现以下任一情况，则回退到手动输入 MAC：

- 使用 `getAddress()` 初始化后解密结果明显错误
- 收到通知但无法解析出合法消息类型
- 初始状态不合法，且多次重试仍失败

### 8.3 可选增强

二期可再补“从广播数据推导 MAC”的自动方案，但不是一期阻塞项。

### 8.4 当前实现细化

当前实现额外补充了两条一期内很关键的细化策略：

- 当 `Scrambler` 没有显式保存普通 `3x3` 打乱的 `cubeState` 时，自动根据打乱公式反推目标状态，用于智能魔方自动判定“已打乱”。
- `MoYu32` 连接时优先使用 Android 返回的设备地址初始化 cipher；如果首轮解析异常，再回退到基于设备名 `WCU_MY32_xxxx` 的默认 MAC 推导。

## 9. Android 15 兼容策略

一期需要从一开始就按新权限模型设计，至少把结构留对。

### 9.1 权限

需要考虑以下运行时权限：

- `BLUETOOTH_SCAN`
- `BLUETOOTH_CONNECT`
- 旧版本条件下的位置权限

### 9.2 扫描 API

当前项目仍使用旧 `startLeScan`。  
一期建议 `MoYu32` 新链路直接基于 `BluetoothLeScanner` 设计，不再依赖旧扫描 API。

### 9.3 当前实现补充（2026-04-05）

- 当前代码已补上 `Android 12+` 所需的 `BLUETOOTH_SCAN`、`BLUETOOTH_CONNECT` 权限声明与运行时申请。
- 当前代码已切换为 `Android 5.0+` 优先使用 `BluetoothLeScanner` 扫描，低版本保留 `startLeScan` 作为兼容兜底。
- 当前工程已提升到 `targetSdkVersion 35`，在 Android 12+/15 上 BLE 扫描主链只请求 `BLUETOOTH_SCAN / BLUETOOTH_CONNECT`；`ACCESS_FINE_LOCATION` 仅保留到 Android 11 及以下的兼容声明。
- 本轮改动已通过 `assembleDebug` 构建验证，但仍需要继续完成 Android 15 真机扫描、重扫、连接与断开回归。

### 9.4 兼容目标

一期的最低要求：

- Android 15 真机上能完成扫描、连接、计时、保存成绩

一期暂不要求：

- 完全重构旧 Giiker/GAN 逻辑到新权限体系

## 10. 计时逻辑约定

一期沿用现有 DCTimer 计时习惯：

1. 用户先完成打乱
2. 用户通过“标记已打乱”进入 ready 状态
3. 首次真实转动开始计时
4. 魔方恢复 solved 状态时自动停表
5. 保存成绩与步序

说明：

- 一期不做 csTimer 那种更复杂的自动识别“打乱结束”机制
- 一期不做训练模式，也不做连续训练
- 这样可以最大程度复用当前 `SmartCube` 与 `MainActivity` 中已有逻辑

补充约定：

- 智能魔方模式下，打乱完成后计时器页面应进入“准备开始但尚未起表”的状态，而不是直接表现为已经开始计时。
- 打乱完成后的第一次还原转动，才应真正触发计时开始。
- 当前实现中最终保存的成绩时间可正确对应真实复原时间，但页面状态表现仍需要继续对齐。

## 11. 数据与持久化

现有结果表已支持 `moves` 字段，一期沿用即可。

一期建议保存：

- time
- scramble
- date
- moves

一期暂不新增：

- 详细设备时间线
- 每步时间数组
- 设备型号字段

这些更适合放到二期复盘能力中统一设计。

## 12. 风险清单

### 12.1 高风险

- Android 端拿到的地址是否能直接用于解密，不确定
- `moyu32` 的 AES 行为如果与 JS 端有字节序差异，调试成本会高

### 12.2 中风险

- 当前项目整体蓝牙结构较旧，新增新链路时容易与旧实现互相影响
- Android 15 权限/扫描流程若处理不严谨，可能出现“能连但扫不到”或“能扫到但无法连接”

### 12.3 低风险

- 成绩保存、步序保存、状态展示部分已有现成基础

## 13. 验证策略

### 13.1 协议层验证

- 能成功扫描并连接 MoYu32
- 能收到 `161 / 163 / 164 / 165`
- `163` 能解出合法 facelet
- `165` 能连续稳定解出 move 和时间差

### 13.2 业务层验证

- 初始状态显示正确
- 电量刷新正确
- 标记已打乱后首转开表
- 完全还原时自动停表
- 保存成绩后数据库内 `moves` 不为空

### 13.3 回归验证

- 原本普通手触计时不受影响
- 原本结果列表、成绩详情、统计不受影响

### 13.4 2026-04-05 真机反馈结论

本轮真机反馈已确认以下内容：

- `MoYu32` 可以被扫描、连接并进入协议解析流程。
- 自动计时主链路已经基本打通，但产品表现仍与目标设计存在差距。

当前已知差距如下：

1. 连接成功后弹出的状态窗口，当前已切到新的立体渲染控件，视角、描边和明暗层次已明显更接近实体魔方；后续仍可继续细调边角圆润感与阴影质感。
2. 计时器页面右下角已切到同一套实时 3D 渲染控件，当前支持单步转层补间动画，并在计时过程中持续刷新；后续仍可继续细调尺寸和动画节奏。
3. 打乱公式区域已支持按正确打乱进度做渐隐/变色提示；当前版本采用颜色区分实现，后续仍可继续迭代为更接近 csTimer 的逐步消隐反馈。
4. 打乱完成后页面已进入 `READY` 状态，并等待第一转才真正起表；当前还额外补了“快打乱时偶发直接起表”的竞态修正，后续可继续根据真机体验细调文案、颜色和观察态表现。
5. 智能魔方计时页面已提供“重置魔方”为还原状态的直接按钮。
6. 智能魔方完成复原后，当前实现已改为直接保存成绩并自动进入下一轮打乱，不再弹出无惩罚 / `+2` / `DNF` 的复核框。

## 14. 一期后续扩展位

如果一期成功，后续可以沿同一路径继续扩展：

- 加入“自动获取 MAC 失败后的广播解析”
- 增加复盘详情与步序展示
- 将 `BluetoothTools` 逐步重构为统一的 BLE 管理层
- 接入更多使用相似 AES 方案的智能魔方设备

## 15. 当前推荐决策

本期采用以下决策：

- 仅支持 `MoYu32 + 3x3`
- 先做协议闭环，不做高级训练功能
- 自行实现 `Moyu32Cipher`，不依赖缺失的 `aes.jar`
- 允许手动输入 MAC
- 新设备按独立协议类接入，不继续把协议写死在旧 `BluetoothTools` 中

这是当前正确性、可执行性、开发成本三者之间最平衡的方案。
