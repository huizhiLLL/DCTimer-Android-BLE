## 当前项目方案主线

当前仍然有效、并且后续还会继续沿用的方案主线，主要有两条：

1. 智能魔方蓝牙计时主线
2. 蓝牙硬件分层与蓝牙计时器接入主线

`Android` 底座升级和 release 打包链路方案已经基本完成，当前更适合作为既定基线保留结论，不再作为独立规划方向维护。

## 一、智能魔方蓝牙计时主线

### 当前目标

- 继续以 `3x3` 智能魔方蓝牙计时为核心方向
- 保持“按打乱进入 `READY` -> 首转起表 -> 还原停表 -> 自动保存”的主流程不回退
- 在现有多协议兼容基础上，继续保持主流程稳定和实现可维护性

### 当前已落地的方案结论

- 智能魔方协议已经不再停留在单一 `MoYu32` 方案，当前主线已覆盖：
  - `MoYu32`
  - `QiYi / Tornado V4 智能版`
  - `GAN v2 / v3 / v4`
- 新协议当前统一收口到 `SmartCubeProtocol`，后续继续扩展新智能魔方时仍应沿用这条分层方式。
- BLE 扫描弹窗已取消旧的协议类别切换，当前以“扫描结果直列 + 连接阶段自动识别协议”为准。
- 智能魔方状态展示当前以自定义 3D 渲染控件为主，不再回到早期静态三面图方案。
- 打乱体验当前以“打乱进度提示 + 偏离后的逆序纠错提示 + 超长纠错时提示先复原 + READY 状态等待首转”为主，不再回退到“连接后直接起表”或“只靠手动观察”的旧思路。
- 在开启 `WCA` 观察时，智能魔方当前按“打乱完成立即进入观察 -> 首次真实转动起表”工作，不再允许通过点击页面手动结束观察起表。
- Android 15 真机上，智能魔方 BLE 扫描、重扫、连接、断开与自动计时主链已继续完成回归。
- 3D 状态图、打乱提示与纠错提示的当前体验已完成一轮细调，现阶段先按现有实现收口。

### 当前仍有效的设计边界

- 重点保障 `3x3` 智能魔方计时主流程，不对外承诺“所有蓝牙魔方都支持”。
- 继续保持“协议层独立类 + `BluetoothTools` 负责扫描/连接/分发”的结构，不在旧大类里继续堆叠协议细节。
- `moves` 步序继续沿用现有数据库字段保存，不在当前阶段提前扩展更重的复盘数据结构。

### 当前主要剩余方向

- 当前智能魔方主线暂无必须继续推进的待补项。
- 如果后续重新确认需要补做异常恢复提示或手动输入 `MAC` 兜底，再作为独立子任务重新立项。

## 二、蓝牙硬件分层与蓝牙计时器接入主线

### 当前目标

- 将原本混在“蓝牙设备”模式下的硬件能力，正式拆成：
  - `智能魔方`
  - `蓝牙计时器`
- 在这条新分层上接入并稳定 `QiYi Smart Timer`

### 当前已落地的方案结论

- `enterTime` 当前已经从旧的单一“蓝牙设备”语义拆成“智能魔方 / 蓝牙计时器”两条独立入口。
- 旧 `SmartTimer.java` 和旧 `BLEDevice.TYPE_GAN_TIMER` 已被清理，不再保留 legacy smart timer 结构。
- `BluetoothTools` 当前已经增加 `SmartTimerProtocol` 分发支路。
- `QiYi Smart Timer` 已按独立 `QiyiSmartTimerProtocol` 接入，不与智能魔方协议类混写。
- 当前蓝牙硬件入口层的职责边界已经明确：
  - `enterTime` 决定用户当前处于哪类硬件模式
  - `BLEDevice.type` 决定连接到的具体协议设备
- BLE 权限首次申请场景下，扫描弹窗应在授权完成后再开启扫描，不再允许“弹窗已打开但首轮扫描早于授权”的空列表体验。
- 当智能魔方 / 蓝牙计时器断联，或者用户在未连上设备前主动关闭连接弹窗时，`enterTime` 应自动回退到普通计时器，避免界面停留在失效硬件模式。
- Android 12 / 15 真机上的 BLE 扫描、连接、断开与自动计时链路已继续完成回归确认。
- 智能魔方与蓝牙计时器两条主链当前已继续确认互不污染。

### 当前仍有效的设计边界

- 智能魔方业务链和蓝牙计时器业务链继续保持分离，不互相复用页面语义。
- 蓝牙计时器模式不显示智能魔方状态图、重置魔方按钮和智能魔方状态弹窗。
- 当前先只正式接入 `QiYi Smart Timer`，不提前铺开更多蓝牙计时器品牌。
- 不恢复旧 `GAN Timer` 兼容分支，避免再次进入双轨维护。

### 当前主要剩余方向

- 真机回归验证 `QiYi Smart Timer` 的 `READY / RUNNING / STOPPED` 主链
- 继续确认断开连接、重连、异常恢复时主页面状态是否稳定
- 在首条蓝牙计时器链路稳定后，再决定是否继续扩更多 timer 协议

## 三、已完成并收口的历史方案

### 1. Android 底座升级

这部分已经从“方案”转为“项目基线”，当前结论如下：

- 已升级到 `AndroidX + AGP 8.9.2 + Gradle 8.11.1 + JDK 17`
- 已将 `compileSdk / targetSdk` 提升到 `35`
- 文件导入导出、打乱导入导出、背景图选择已迁移到 `SAF / Uri`
- 旧 `FileSelectorDialog`、APK 自更新入口、安装权限与 `FileProvider` 已退出主线
- BLE 权限链当前按 Android 新权限模型和定位兼容模式收口

后续除非再次出现平台级升级需求，否则不再单独维护“底座升级方案”文档。

### 2. Release 打包配置

这部分也已经从阶段任务转为既定发布基线，当前结论如下：

- release 签名入口已经标准化
- 支持 `key.properties` 和 `-P` 参数两种签名注入方式
- 未提供签名信息时会直接报错，不再生成不可分发包
- 当前版本号基线已进入 `2.0 Beta`
- 已保留 `rel.bat` 作为本地 release 构建快捷入口

后续除非发布链再变动，否则不再单独维护方案文档。

### 3. WCA 观察提醒

这部分当前也已经形成既定实现约束，当前结论如下：

- `WCA` 观察模式保留现有 `8s / 12s` 阈值触发，不改动原本 `15s -> +2 / 17s -> DNF` 判定链。
- 观察提醒当前优先播放项目内置音频素材，默认查找：
  - `app/src/main/assets/inspection/inspection_8s.wav`
  - `app/src/main/assets/inspection/inspection_12s.wav`
- 如果未提供上述素材，应用会自动回退到系统 `TTS` 播报 `8秒 / 12秒`，避免功能直接失效。

后续如果要继续扩展男声 / 女声 / 音量配置，优先在现有素材播放链路上追加，不回退到重新改造观察状态机。
