# DCTimer-Android 架构边界

日期：`2026-05-16`

## 文档职责

本文只维护架构边界、模块职责和依赖方向；当前完成度放在 `docs/project.md`，后续计划和已收口决策放在 `docs/roadmap.md`。

## 总体边界

项目以“硬件类型分层、协议独立、页面语义隔离”为主要约束。智能魔方和蓝牙计时器都经由 BLE 接入，但业务含义不同，不混用状态机、页面入口或保存逻辑。

## 硬件入口边界

- `enterTime` 表示用户当前选择的计时方式和硬件模式，负责页面语义。
- `BLEDevice.type` 表示实际连接到的协议设备类型，负责连接后的协议分发和能力识别。
- 两者不能互相替代：入口模式解决“用户在哪条业务链”，设备类型解决“连接对象是什么协议”。

## 蓝牙协议边界

### 智能魔方

- 智能魔方协议统一收口到 `SmartCubeProtocol`。
- 新设备按独立协议类接入。
- `BluetoothTools` 负责扫描、连接、基础分发和协议识别，不继续堆叠具体协议细节。
- 具体协议类负责解析设备数据、状态同步、电量同步、转动上报和必要 ACK。
- 单个 BLE 通知内可能携带多个转动事件，协议层必须优先完整解析后再进入 history / facelet 兜底恢复。
- history 补回的缺失转动如果没有设备时间戳，应通过前后真实时间戳或本地触发时间估算，避免固定极小 delta 影响成绩。

### 蓝牙计时器

- 蓝牙计时器协议统一收口到 `SmartTimerProtocol`。
- 新 timer 品牌按独立 timer 协议类扩展，不复用智能魔方协议结构。
- 不恢复旧 `GAN Timer` 兼容分支，避免重新进入双轨维护。

## 业务链隔离

### 智能魔方链路

- 页面可以展示智能魔方 3D 状态图、重置魔方入口、打乱进度和纠错提示。
- 计时语义以“打乱完成后等待首转，首个真实转动起表，复原停表”为主。
- `WCA` 观察可参与智能魔方流程，但起表只能由真实转动触发。
- 成绩保存时可保存 `moves` 和 `solve_meta`。

### 蓝牙计时器链路

- 页面不显示智能魔方状态图、重置魔方按钮和智能魔方状态弹窗。
- 计时语义遵循设备自身状态：`READY / RUNNING / STOPPED`。
- 准备态只显示待起表，不提前跑表。
- 运行态可以由本地时钟刷新显示，停表后以设备成绩为准。
- 蓝牙计时器模式不启用应用内 `WCA` 观察和观察提示。

## 数据边界

- `moves` 作为成绩对外展示字段保存。
- `solve_meta` 为可空追加字段，兼容旧库升级和旧数据导入。
- 历史 `PB` 标记是由当前分组成绩按时间顺序重新计算得到的统计派生状态，不写入数据库，也不改变成绩表顺序。
- 当前重建统计先只面向 `333` 智能魔方 CFOP 分段。

## UI 与渲染边界

- 普通计时、智能魔方、蓝牙计时器可以共享基础计时页，但硬件语义必须由入口模式控制。
- 智能魔方状态预览使用独立 `SmartCube3DView`，内部基于 `GLSurfaceView + OpenGL ES 2.0` 渲染 `3x3` 魔方，并允许手动拖动改变视角。
- `SmartCube3DView` 只消费业务层传入的 `facelet` 状态和 `move` 动画指令，不直接接触 BLE 协议解析、计时状态机或成绩保存逻辑。
- 普通打乱图继续使用 `SmartCubeImageView` 的 Canvas / Bitmap 路径。
- 智能魔方和蓝牙计时器绿色准备态可复用沉浸式预览布局；普通计时器 `READY` 保持普通计时区布局，只在 `RUNNING` 后进入沉浸式计时布局。

## 平台边界

- Android 工程基线保持 `AndroidX + AGP 8.9.2 + Gradle 8.11.1 + JDK 17`。
- `compileSdk / targetSdk` 保持 `35`，除非出现新的平台升级任务。
- 文件访问继续使用 `SAF / Uri`，不回退到旧文件路径选择器。
- release 包必须提供签名信息；缺少签名信息时应直接失败。
