# DCTimer-Android 架构边界

日期：`2026-05-02`

## 总体边界

当前架构以“硬件类型分层、协议独立、页面语义隔离”为主要约束。智能魔方和蓝牙计时器都经由 BLE 接入，但业务含义不同，不应因为同属蓝牙设备而混用状态机、页面入口或保存逻辑。

## 硬件入口边界

- `enterTime` 表示用户当前选择的计时方式和硬件模式。
- `BLEDevice.type` 表示实际连接到的协议设备类型。
- `enterTime` 负责区分“智能魔方 / 蓝牙计时器 / 普通计时器”的页面语义。
- `BLEDevice.type` 负责连接后的协议分发和设备能力识别。

这两个字段不能互相替代：入口模式解决用户当前在哪条业务链，设备类型解决连接对象是什么协议。

## 蓝牙协议边界

### 智能魔方

- 智能魔方协议统一收口到 `SmartCubeProtocol`。
- 新智能魔方协议继续使用独立协议类接入。
- `BluetoothTools` 负责扫描、连接、基础分发和协议识别，不继续堆叠具体协议细节。
- 具体协议类负责解析设备数据、状态同步、电量同步、转动上报和必要 ACK。

### 蓝牙计时器

- 蓝牙计时器协议统一收口到 `SmartTimerProtocol`。
- `QiYi Smart Timer` 使用独立 `QiyiSmartTimerProtocol`。
- 不恢复旧 `GAN Timer` 兼容分支，避免重新进入双轨维护。
- 后续接入更多 timer 品牌时，按新协议类扩展，不复用智能魔方协议结构。

## 业务链隔离

### 智能魔方链路

- 页面可以展示智能魔方 3D 状态图、重置魔方入口、打乱进度和纠错提示。
- 计时语义以“打乱完成后等待首转，首个真实转动起表，复原停表”为主。
- `WCA` 观察可参与智能魔方流程，但起表只能由真实转动触发。
- 成绩保存时可保存 `moves` 和 `solve_meta`。

### 蓝牙计时器链路

- 页面不显示智能魔方状态图、重置魔方按钮和智能魔方状态弹窗。
- 计时语义遵循设备自身状态：`READY / RUNNING / STOPPED`。
- 准备态只显示待起表，不提前跑表。
- 运行态可以由本地时钟刷新显示，停表后以设备成绩为准。
- 蓝牙计时器模式不启用应用内 `WCA` 观察和观察提示。

## 数据边界

- `moves` 继续作为成绩的对外展示字段保存。
- 重建统计元信息使用可空 `solve_meta` 字段追加，保证旧库升级和旧数据导入兼容。
- 成绩列表的历史 `PB` 标记是由当前分组成绩按时间顺序重新计算得到的统计派生状态，不写入数据库，也不改变成绩表结构。
- 当前重建统计先只面向 `333` 智能魔方 CFOP 分段。
- 真机未验证的重建能力，不能作为已完成能力写入产品说明。

## UI 与状态边界

- 普通计时、智能魔方、蓝牙计时器可以共享基础计时页，但硬件语义必须由入口模式控制。
- 普通成绩列表允许按单次、滚动平均 1、滚动平均 2 的历史 `PB` 记录临时排序；该排序只影响列表展示，不改变成绩保存顺序。
- 绿色准备态可复用沉浸式预览布局，但不同模式的起表来源必须保持独立。
- 设备断联或连接弹窗未连接即关闭时，应回退到普通计时器，避免页面停留在失效硬件模式。
- BLE 权限首次授权完成后，应重新进入扫描流程，避免出现首轮空列表。

## 平台与发布边界

- Android 工程基线保持 `AndroidX + AGP 8.9.2 + Gradle 8.11.1 + JDK 17`。
- `compileSdk / targetSdk` 保持 `35`，除非出现新的平台升级任务。
- 文件访问继续使用 `SAF / Uri`，不回退到旧文件路径选择器。
- release 包必须提供签名信息；缺少签名信息时应直接失败，不生成不可分发包。
