# MoYu32 智能魔方计时一期任务拆分

## 1. 任务目标

围绕 `MoYu32 + 3x3`，按“先协议、再接业务、最后收体验”的顺序完成第一期接入。

预期结果：

- Android 15 真机可以扫描并连接 MoYu32
- 能稳定解析状态、电量、转动
- 能完成“标记已打乱 -> 首转开表 -> 还原停表 -> 保存成绩”
- 成绩可写入现有结果体系，并保留步序

### M1. 设备与权限底座

目标：

- 建立 Android 12+ 到 Android 15 可用的 BLE 扫描与连接最小链路
- 为 MoYu32 提供单独的扫描/连接入口

任务：

- 梳理并补齐蓝牙权限申请流程
- 增加 `MoYu32` 设备类型与 UI 入口
- 基于 `BluetoothLeScanner` 建立扫描链路
- 支持按设备名前缀 `WCU_MY3` 过滤目标设备
- 完成连接、服务发现、读写特征绑定

产出：

- 可扫描到 MoYu32 设备
- 可完成连接和断开
- 日志可输出服务、特征和连接状态

验证点：

- Android 15 真机上授予权限后能看到设备
- 点选设备后能连上，不闪退
- 断开后 UI 状态恢复正常

### M2. AES 与协议打通

目标：

- 完成 `moyu32cube.js` 对应的 Android 版加解密与消息解析

任务：

- 提取并固化 MoYu32 的 key/iv 常量
- 实现 `Moyu32Cipher`
- 优先尝试使用 `BluetoothDevice.getAddress()` 初始化 cipher
- 设计手动输入 MAC 的兜底交互
- 发送 `161 / 163 / 164` 请求
- 解析 `161 / 163 / 164 / 165`

产出：

- 可打印合法消息类型
- 可得到初始状态 facelet
- 可得到电量值
- 可得到转动序列与时间偏移

验证点：

- `163` 能得到合法 54 位 facelet
- `164` 电量值在合理区间
- 真实拧动时持续收到 `165`
- 手动输入 MAC 后协议恢复正常

### M3. SmartCube 事件接入

目标：

- 把 MoYu32 的协议事件接入现有 DCTimer 智能魔方计时主链路

任务：

- 定义统一事件回调：
  - 初始化状态
  - 电量更新
  - 单步转动
  - 协议错误
- 将单步转动同步给 `SmartCube`
- 接入“当前状态是否等于 scramble state”的判断
- 接入“当前状态是否等于 solved state”的判断
- 保持与现有 `Result.insert(..., SmartCube cube)` 兼容

产出：

- `SmartCube` 能接收 MoYu32 的 move
- 步数、步序、总时间可以正确累计

验证点：

- 随机打乱后，手动标记已打乱可进入 ready
- 首转时开始计时
- 完全还原后停止计时
- 成绩保存后数据库 `moves` 不为空
- 普通 `3x3` 公式打乱在未显式保存 `cubeState` 时，仍可正确自动判定目标状态

### M4. 交互收尾

目标：

- 让一期功能达到可日常使用状态

任务：

- 设备选择弹窗加入 `MoYu32`
- 当自动获取 MAC 失败或解密失败时弹出 MAC 输入框
- 在状态弹窗中显示当前状态与电量
- 将状态弹窗中的魔方状态图升级为尽可能接近 csTimer 的 3D 魔方渲染图
- 将计时器页面右下角的打乱状态图升级为 3D 渲染图
- 在计时器页面直接提供“重置为还原状态”按钮
- 让打乱完成后进入 ready/观察状态，而不是直接表现为已开始计时
- 让打乱公式区域参考 csTimer：随着正确打乱推进，已完成的打乱字母逐渐消失
- 以上打乱推进逻辑固定以“白上绿前”方向为基准
- 补齐必要提示文案：
  - 连接失败
  - 解密失败
  - MAC 无效
  - 已断开

产出：

- 一期功能可由普通用户按流程使用

验证点：

- 异常路径有提示，不会无响应
- 手动输入 MAC 后可继续连接或重新初始化
- 状态弹窗可正确显示接近 csTimer 风格的 3D 魔方状态
- 计时器页面右下角可显示 3D 打乱状态图
- 打乱完成后页面进入 ready/观察状态，而不是直接起表
- 已完成的正确打乱步骤会在公式区逐渐消失
- 计时器页面存在“重置为还原状态”按钮

## 4. 每阶段验收标准

### 阶段 A：协议可读

完成标准：

- 已连接设备
- 已成功读取并解析至少一条 `163`
- 已在日志中打印出合法初始状态

未完成表现：

- 只连接成功但无法解出任何合法数据

### 阶段 B：转动可追踪

完成标准：

- 用户每转一步，应用都能稳定收到 move 事件
- move 顺序与实际手法一致
- move 对应时间差大体合理

未完成表现：

- 有漏步、重复步、顺序反了、时间差明显异常

### 阶段 C：计时可闭环

完成标准：

- 标记已打乱后首转开表
- 还原后停表
- 成绩与步序成功保存
- 打乱完成后页面进入未起表的 ready/观察状态
- 第一次还原转动才真正开始计时显示

未完成表现：

- 能收步但无法正确开停表

### 阶段 D：体验可使用

完成标准：

- 连接流程完整
- 失败时有提示
- 手动 MAC 兜底可用
- 状态弹窗中的魔方状态图足够接近 csTimer 的 3D 视图
- 计时器页面右下角状态图为 3D 视图
- 打乱推进反馈、ready 状态和重置按钮都可直接在计时器页使用

未完成表现：

- 遇到异常只能看日志，用户无法自行恢复
- 必须依赖隐藏入口才能看状态或重置魔方

## 5. 风险拆分与应对

### 风险 1：`getAddress()` 不能直接用于解密

应对：

- 提前预留手动输入 MAC 的回退交互
- 将 cipher 初始化与连接流程解耦，允许连接后重新输入 MAC 并重试初始化

### 风险 2：JS 与 Java AES 实现字节处理不一致

应对：

- 在协议层加入原始字节日志
- 对照 `moyu32cube.js` 中 `decode` 逐步比对每一块输出
- 先验证 `161 / 163`，再验证 `165`

### 风险 3：Android 15 权限链路阻塞

应对：

- 将权限与 BLE 扫描初始化前置
- 把权限失败、拒绝授权、永久拒绝分开处理

### 风险 4：旧蓝牙代码互相干扰

应对：

- MoYu32 先走新链路
- 不在一期强行复用所有旧逻辑
- 保持旧 Giiker/GAN 入口不动，避免回归风险扩大

## 6. 建议开发节奏

推荐按以下节奏推进：

1. 第一次提交只做底座与扫描连接
2. 第二次提交只做 AES 与协议解析
3. 第三次提交只做 SmartCube 接入与计时闭环
4. 第四次提交补交互、提示与文档修正

这样每一步都能独立验收，也方便定位问题。

## 8. 一期完成定义

满足以下条件即可判定一期完成：

- 用户可在 Android 15 手机上连接 MoYu32
- 用户可通过 DCTimer 完成一次完整 3x3 智能魔方计时

- 自动 MAC 失败时可手动输入 MAC 继续使用
- 整体流程无需依赖 csTimer 或外部网页辅助
